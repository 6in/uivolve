# PMI 評価レポート: uivolve 方式 vs Figma

対象: 「ExtJS 互換 DSL による画面モック + AI への実装引き渡し」という uivolve のアプローチを、
画面モック作成の主流ツールである Figma と比較して PMI 方式 (Plus / Minus / Interesting) で評価する。

- 作成日: 2026-07-08
- 前提: 主な用途を**業務システムの画面モック作成と実装への引き渡し**と置く。
  ビジュアルデザイン全般 (ブランディング、LP、モバイルアプリの外観) は uivolve のターゲット外。

---

## Plus (優位点)

### P1. テキストであること — Git に完全に乗る
DSL はただのテキストなので、diff・ブランチ・PR レビュー・マージ・blame がそのまま使える。
「この画面、先週から何が変わった?」に `git diff` で答えられる。
Figma にも版履歴はあるが、コードレビューのワークフローには統合できない。

### P2. AI 親和性 — 「AI に渡す」ことを最初から設計に含む
LLM はピクセルより構造化テキストを高忠実に読み書きする。DSL + itemId + JSON Schema により
「`orderGrid` に列を追加して」「このモックを React + MUI で実装して」が精度高く通る。
Figma も MCP / Dev Mode で AI 連携を進めているが、デザインデータからの構造抽出は
ノイズが多く、レイヤー名の規律に依存する。

### P3. 作成速度 — エンジニアはキーボードの方が速い
xtype スニペット + 補完 + ライブプレビューで、業務画面の定番構成 (border + グリッド + フォーム)
なら数分で組める。図形のドラッグ・整列・グループ化より、宣言的 config の方が
エンジニアには速く、修正も一括置換が効く。

### P4. 「本物の HTML」で動くモック
折りたたみ・タブ切替・スプリットバーのドラッグ・行選択が実際に動く。
ブラウザの実レンダリングなので、スクロールやオーバーフローの挙動が本物。
Figma のプロトタイプは画面リンク遷移が中心で、コンテナ挙動の再現は作り込みが必要。

### P5. 仕様書との一体化 (MDX)
Markdown 仕様書の中にモック定義が埋まるので、**仕様と絵が構造的に乖離しない**。
仕様変更 = DSL 変更 = モック更新が 1 コミット。Figma では仕様書とデザインが
別システムに分かれ、ドキュメントドリフトが起きやすい。

### P6. コストと配布
OSS・無料・ブラウザのみで完結 (GitHub Pages で配布済み)。Figma はシート課金で、
閲覧以上の関与には編集権限の管理が必要。

## Minus (劣位点)

### M1. ビジュアルデザインの自由度がない
ピクセル単位の自由な表現・ブランディング・イラスト・独自ビジュアルは不可能。
「業務画面の定番部品の組み合わせ」から出ると急に無力になる。
Figma はあらゆる見た目を作れる汎用キャンバス。

### M2. 非エンジニアの参加障壁
DSL (テキスト) を書けない職種には作成側として参加しにくい。
Figma の直感的操作・リアルタイム共同編集・コメント・FigJam のような
「デザイナー/PM/ステークホルダーを巻き込む」文化装置がない。
レビューは Git/PR 経由になり、開発者以外には敷居が高い。

### M3. デザインシステム・エコシステムの差
Figma にはデザイントークン、コンポーネントライブラリ、バリアント、プラグイン、
コミュニティ資産という巨大なエコシステムがある。uivolve のテーマ機構 (CSS 変数) は
その代替にはならない。

### M4. 「完成イメージ」の提示力が弱い
テーマは ExtJS 風の見た目に固定される。最終プロダクトのビジュアル承認を取る場面では
「これはあくまで構造のモックです」という注釈が必要になり、Figma のハイファイデザインが持つ
説得力・ワクワク感はない。

### M5. 成熟度
POC 段階。コンポーネント数 (23 xtype)・エッジケース・ドキュメント・コミュニティの
すべてが発展途上。Figma は 10 年級の成熟プロダクト。

## Interesting (興味深い点)

### I1. Design-as-Code への収斂
Figma 自身が Dev Mode / Code Connect / MCP で「デザイン→コード」に接近している。
uivolve は逆方向 (コード→デザイン) から同じ地点に向かっており、
「デザインの正準表現はコードになる」という業界トレンドの一つの実装と言える。

### I2. モックの主目的が「人に見せる」から「AI に指示する」へ
従来のモックは人間 (実装者・承認者) への伝達手段だった。実装者が AI になると、
**絵よりも構造化テキストの方が忠実に伝わる**ため、評価軸そのものが反転する。
この反転が起きた世界では、Figma→実装の変換損失こそがボトルネックになる。

### I3. ExtJS 互換という語彙の選択
20 年分の業務 UI パターンの語彙 (xtype / fieldLabel / border layout) を再利用しており、
これは **LLM の学習データに大量に含まれる語彙**でもある。AI が「知っている言葉」で
モックを書かせる設計は、独自 DSL を発明するより AI 出力の精度が出やすい可能性がある。

### I4. 競合ではなく住み分け・併用が自然
- ビジュアルデザイン / 顧客向け提案 → Figma
- 業務画面の構造仕様 / 開発チーム内の合意 / AI への引き渡し → uivolve
さらに DSL→Figma (API 生成) や Figma→DSL (構造抽出) の変換も理論上可能で、
ブリッジを作れば対立軸自体が消える。

### I5. 「動く仕様書」という文化への影響
MDX + PR 運用は、仕様レビューをコードレビューと同じフローに載せる。
仕様書が「読むだけの文書」から「動いて検証できる成果物」になるのは、
ツール比較を超えたプロセス変化の入り口かもしれない。

### I6. AI との往復ループの速さ
AI が DSL を生成 → 即プレビュー → 人が微修正 → また AI へ、のループが
テキストベースゆえに秒単位で回る。JSON Schema による出力検証も挟める。
このループ速度は「AI とペアでモックを練る」という新しい作業形態を可能にする。

## 総合評価

| 観点 | uivolve | Figma |
|---|---|---|
| 業務画面モックの作成速度 (エンジニア) | ◎ | ○ |
| ビジュアルデザインの自由度 | × | ◎ |
| Git / PR / CI との統合 | ◎ | △ |
| AI への引き渡し精度 | ◎ | △〜○ (改善中) |
| 非エンジニアの参加しやすさ | △ | ◎ |
| 動くモック (実 HTML 挙動) | ◎ | ○ (作り込み次第) |
| エコシステム / 成熟度 | △ | ◎ |
| コスト | ◎ (無料) | ○ (シート課金) |

**結論**: Figma の代替ではなく、**「エンジニア主導の業務システム開発における、
構造仕様としてのモック + AI 実装連携」という Figma が弱いニッチ**を突くツール。
勝ち筋は「デザインツール」として戦わないこと。
「動く仕様書」「AI に渡せるモック」という別のカテゴリ名で語るのが良い。

### 補記: エンタープライズ業務画面では「自由度のなさ」が利点になる

M1 (ビジュアル自由度がない) は、エンタープライズ業務画面のドメインでは反転して強みになる。

1. **統一性が正義** — 数十〜数百画面の業務システムでは、画面ごとにデザインが違うことが悪。
   部品が固定なら誰が作っても同じ操作感になる。
2. **利用者の学習コスト** — 毎日長時間使うオペレーターには、見た目の新鮮さより
   「グリッドは全部同じ操作」の方が価値がある。
3. **議論が本質に集中する** — 部品が固定なら、レビューは「ボタンの色」ではなく
   「項目の過不足・導線・権限」の話になる。仕様書に必要な情報だけが残る。
4. **実装との 1:1 対応** — ハイファイデザインのような解釈の余地 (= 実装時の変換損失) がない。

SAP Fiori / Salesforce Lightning / Oracle APEX / そして ExtJS 自身と、エンタープライズは
もともと「決まった部品の宣言的組み立て」の思想で動いてきた。uivolve はその思想を
ライセンスフリー + AI 時代向けに蒸留したものであり、このドメインでは Figma の自由な
キャンバスの方がむしろ「統一性を壊す自由」を与えてしまう、という逆説が成り立つ。
